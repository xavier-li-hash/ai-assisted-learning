## 背景：为什么会有Lambda架构

2011年，Nathan Marz（Twitter工程师）提出Lambda架构，解决一个根本矛盾：

```
实时性   vs   准确性

流处理：低延迟（秒级），但容易有误差（数据乱序、重复）
批处理：高准确性（T+1），但延迟高
```

Lambda架构的解法：**两条腿走路，查询时合并结果**。

---

## Lambda架构三层

```
原始数据
    │
    ├──→ 批处理层（Batch Layer）   → 准确，慢（Spark/Hive，T+1）
    │        │
    │        ↓
    │    批处理视图（Batch View）
    │
    ├──→ 速度层（Speed Layer）     → 快，可能有小误差（Flink/Storm）
    │        │
    │        ↓
    │    实时视图（Realtime View）
    │
服务层（Serving Layer） ← 合并两个视图，响应查询
    │
    ↓
最终结果（查询时 = 批处理视图 + 实时视图合并）
```

## Lambda架构的缺点（必须知道）

| 缺点 | 说明 |
|------|------|
| 代码重复 | 同一业务逻辑要写两遍（批+流） |
| 一致性难保证 | 批处理和实时视图的结果可能不一致 |
| 运维复杂 | 两套系统独立运维，成本高 |
| 数据修复麻烦 | 批处理出错需要重跑，实时层怎么办？ |

## Kappa架构预告（明天）

Kappa架构的核心思路：**用流处理统一批处理**

```
不搞两套系统，只有一条流：
原始数据 → Kafka（永久保存）→ Flink流处理 → 服务层

当需要重算历史数据时：从Kafka最早的offset重放即可
```
